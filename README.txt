============================================================
 共有ドライブ ビジュアルポータル
============================================================

共有ドライブのフォルダ構造を階層マップで可視化し、
ユーザー単位でのアクセス権限を横断検索できるWebアプリです。

データ更新はGoogleスプレッドシートを編集するだけ。
GitHub Actions が自動で反映します。


============================================================
 画面構成
============================================================

  portal.html    … 共有ドライブ一覧 (エントリポイント)
                   ドライブ名・フォルダ名で横断検索できます
  drive.html    … 個別ドライブのフォルダツリー詳細マップ
  search.html   … メールアドレスでアクセス権限を横断検索

各画面はナビゲーションタブで相互に行き来できます。


============================================================
 操作Tips
============================================================

◆ ポータル画面 (portal.html)
  - 検索欄にドライブ名入力 → リスト絞り込み + 候補補完
  - 検索欄にフォルダ名入力 → マッチしたフォルダ一覧が下に表示
  - フォルダ行クリック → 該当ドライブの詳細マップが開き、
                        そのフォルダに自動ジャンプ

◆ ドライブ詳細画面 (drive.html)
  - フォルダ名クリック → Google Drive を新タブで開く
  - 「i」アイコンホバー → アクセス権限ユーザー一覧を表示
  - 「i」アイコンクリック → 子フォルダを折りたたみ／展開
  - 上部の検索ボックス → メアドで権限保持者を絞り込み

◆ 権限検索画面 (search.html)
  - メアド部分一致で全ドライブ・全フォルダを横断検索
  - 「詳細マップを開く」 → drive.html が該当ユーザーで自動絞り込み


============================================================
 運用方法
============================================================

本アプリは2つの運用モードをサポートしています。


◆◆ モード1: GitHub Pages + Google Sheets 自動連携 (推奨) ◆◆

  スプレッドシートを更新するだけで、GitHub Actions が自動で
  data.js を再生成し、公開サイトに反映します。
  エンドユーザーはURLを開くだけで常に最新データを閲覧できます。

  セットアップ手順は SETUP_GITHUB.md を参照してください。

  日々の運用:
    1. 管理者: スプレッドシートを更新
    2. 待つ: 翌朝9時 (JST) に自動反映
       (即時反映したい場合は GitHub Actions タブで Run workflow)
    3. エンドユーザー: 公開URLを開くだけ


◆◆ モード2: ローカル配布 (ZIP配布) ◆◆

  フォルダごとZIP化して配布する方式。Python実行環境が
  必要な代わりに、社内限定で運用できます。

  セットアップ:
    1. Python 3.8+ をインストール
    2. pip install pandas openpyxl
       (Sheets を使う場合は追加で) pip install gspread google-auth

  日々の運用:
    1. 「共有ドライブフォルダ構成CSV.xlsx」を最新版に置き換え
       (または build_config.json を Sheets 設定に変更)
    2. このフォルダ内でコマンド実行:
         python build.py
    3. data.js が再生成される
    4. フォルダごとZIP化して配布
    5. エンドユーザー: portal.html をダブルクリック

============================================================
 データ構造 (参考)
============================================================

build.py はデータソースの「出力結果_YYYYMMDD_HHMMSS」シートを読み、
以下の構造を data.js に出力します:

  window.DRIVE_DB = {
    meta: { generated_at, drive_count, folder_count, user_count },
    drives: [
      { id, name, folder_count, user_count, svg_height,
        folders: [
          { id, name, level, parent, url, users[], layout: {x, y, w} }
        ]
      }
    ],
    users: {
      "email@example.com": [ { drive_id, folder_id, role } ]
    }
  }

処理時間目安: 30秒〜1分 (ドライブ数200件・約7,600フォルダの場合)


============================================================
 トラブルシューティング
============================================================

Q. portal.html を開いても「data.js を読み込めませんでした」と出る
A. data.js が同じフォルダにあるか確認してください。
   なければ python build.py を実行して生成してください。
   GitHub Pages運用の場合は Actions のビルドが成功しているか確認。

Q. ドライブ詳細を開くと真っ白
A. ブラウザのコンソール (F12) を確認してください。

Q. GitHub Actions のビルドが失敗する
A. SETUP_GITHUB.md のトラブルシューティング節を参照してください。

Q. 文字化けする
A. ブラウザのエンコーディング設定が UTF-8 になっているか確認。


============================================================
 セキュリティ備忘録
============================================================

- credentials.json は絶対に git commit しないこと (.gitignoreで保護)
- もし誤ってcommitしたら、GCPでキーを失効 → 再発行
- サービスアカウントは「閲覧のみ」権限で十分
- GitHub Pagesは Public なので、data.js の中身は全世界公開前提
  (メールアドレス・フォルダ名・ドライブURL等が含まれる)
